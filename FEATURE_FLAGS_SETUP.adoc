== Overview

This installation document provides comprehensive guidance for integrating CloudBees Unify Feature Management into applications. The document uses a theme park web application built with **Go** (backend) and **React** (frontend) as a reference implementation, demonstrating feature flag patterns from basic boolean toggles to advanced targeting and experimentation.

=== What is Feature Management?

Feature Management is a software development practice that decouples code deployment from feature release. Instead of deploying code to release features, teams deploy code with features hidden behind configuration flags, then reveal features through the CloudBees Unify platform.

**Key Benefits:**

* **Risk Reduction**: Test in production with limited exposure
* **Faster Recovery**: Instant rollback without redeployment
* **Experimentation**: A/B testing and gradual rollouts
* **Personalization**: Target features to specific users
* **Operations Control**: Kill switches for problematic features

== Architecture

This document demonstrates feature management integration using a distributed web application:

**Architecture Components:**

* **Frontend (React + JavaScript)**: UI layer that displays feature flag results from backend API
* **Backend (Go)**: Server-side feature flag evaluation for business logic
* **CloudBees Unify Platform**: Centralized flag management and configuration
* **SDK Communication**: Real-time flag updates via Server-Sent Events (SSE) with 60-second polling backup

NOTE: This tutorial uses a **backend-only SDK integration pattern**. The Go backend evaluates feature flags and returns results via REST API. The frontend does not include the CloudBees SDK. This is the recommended pattern for server-side applications as documented in the https://docs.cloudbees.com/docs/cloudbees-feature-management-api/latest/[CloudBees SDK Reference].

== Requirements

=== Prerequisites

==== CloudBees Platform Requirements

* **Active CloudBees Unify account**
* **CloudBees Unify Edition 2** (Feature Management is not available in Edition 1)
* **Feature Management enabled** for your organization
* **Appropriate permissions** via role assignment:
** **Admin role** (recommended for initial setup - provides full access), OR
** **Custom role** with at minimum:
*** Flag: Create, Update (feature flags management)
*** Environment: Create, Update (environment management)
*** Custom property: Create, Update (user-defined attributes)
*** Target group: Create, Update (user segmentation)

==== Technical Requirements

* **Supported SDK language**: Your application must use one of the following languages:
** **Client-side SDKs**: JavaScript (browser), React Native, Android, iOS/tvOS
** **Server-side SDKs**: Go, Java, Node.js, Python, Ruby, PHP, .NET/C#, C/C++
** See https://docs.cloudbees.com/docs/cloudbees-unify/latest/install-sdk/[SDK installation guide] for full list
* **SDK version 6.x or higher** (do not use SDK versions prior to 6.x)
* **Network connectivity**: Outbound HTTPS access to CloudBees domains:
** `https://cloudbees.io` (UI and authentication)
** `https://api.cloudbees.io` (API requests)
** `https://rox-state.cloudbees.io` (flag state)
** `https://rox-conf.cloudbees.io` (flag configuration)
** `https://fm-analytics.cloudbees.io` (analytics)
** `https://sdk-notification-service.cloudbees.io/sse` (real-time updates via Server-Sent Events)
* **Firewall configuration**: If your organization uses network allowlists/firewalls, the domains above must be allowlisted for outbound traffic

[NOTE]
====
**For locked-down/private network environments**: CloudBees offers single-tenant deployments to establish secure private connections. Contact CloudBees sales if your SCM, CI/CD, or deployment environments are on private networks.
====

[NOTE]
====
**Security and compliance**: CloudBees Feature Management is SOC 2 Type II compliant. The platform does not have access to or store end-user PII (personally identifiable information). Flag evaluations occur client-side using locally available attributes.
====

=== Task 2 - Add Feature Flags and Initialize SDK

Now you'll add the SDK imports and implement feature flags in your application.

==== Step 1 - Add SDK Imports

Edit `backend/main.go`. Find the import section at the top of the file. It currently looks like:

[source,go]
----
package main

import (
	"log"
	"net/http"

----

Add the CloudBees SDK and godotenv imports:

[source,go]
----
package main

import (
	"log"

)
----

**What we added:**
* `"os"` - to read environment variables (for FM_KEY)
* `"time"` - for SDK timeout handling
* `"github.com/joho/godotenv"` - to load `.env` file
* `"github.com/rollout/rox-go/v5/server"` - CloudBees SDK

Save the file.

Now that the imports are in your code, tidy up dependencies:

[source,console]
----
$ go mod tidy
----

This ensures go.mod is consistent with your imports. Since godotenv is now imported in `main.go`, it will remain in go.mod.

==== Step 2 - Add Flag Structure

Still in `backend/main.go`, scroll down to find the `func main()` function. You'll add the flag definitions right before it.

Your file currently looks like this:

[source,go]
----
import (
	"log"
	"net/http"
	"os"
	"time"

	"github.com/joho/godotenv"
	"github.com/rollout/rox-go/v5/server"
)

func main() {
    config := LoadConfig()
    // ... rest of main
}
----

Add the flag structure **between the imports and the main() function**:

[source,go]
----
import (
	"log"
	"net/http"
	"os"
	"time"

	"github.com/joho/godotenv"
	"github.com/rollout/rox-go/v5/server"
)

type Flags struct {
    ShowNewEvent   server.RoxFlag
    EnableVipPass  server.RoxFlag
    TicketUpsells  server.RoxFlag
    EnablePromos   server.RoxFlag
}

var flags = &Flags{
    ShowNewEvent:   server.NewRoxFlag(false),
    EnableVipPass:  server.NewRoxFlag(false),
    TicketUpsells:  server.NewRoxFlag(true),
    EnablePromos:   server.NewRoxFlag(true),
}

func main() {
    config := LoadConfig()
    // ... rest of main
}
----

**What this code does:**

* `type Flags struct` - Defines a container for all your feature flags
* `server.RoxFlag` - CloudBees boolean feature flag type
* `server.NewRoxFlag(defaultValue)` - Creates a flag with a fallback value
* Default values are used if CloudBees connection fails or times out

Save the file.

==== Step 3 - Add SDK Initialization Code

Now modify the `main()` function to initialize the CloudBees SDK at startup.

Find the existing `main()` function in `backend/main.go`. It currently looks like:

[source,go]
----
func main() {
    config := LoadConfig()

    app := NewApp(config)

    server := &http.Server{
        Addr:    ":" + config.Port,
        Handler: withCORS(config.AllowedOrigin, app.routes()),
    }

    log.Printf("Theme Park API listening on %s", server.Addr)
    log.Printf("API endpoints: /api/health, /api/events, /api/promotions, /api/tickets/purchase")
    if err := server.ListenAndServe(); err != nil {
        log.Fatal(err)
    }
}
----

Replace it with:

[source,go]
----
func main() {
    // Load .env file for local development
    _ = godotenv.Load()

    config := LoadConfig()

    // Initialize CloudBees SDK with impression handler
    options := server.NewRoxOptions(server.RoxOptionsBuilder{
        ImpressionHandler: func(e server.ImpressionArgs) {
            // Log impression data for debugging
            log.Printf("Impression: flag=%s, value=%v", e.ReportingValue.Name, e.ReportingValue.Value)
            // In production, you could send this to analytics or monitoring tools
        },
    })
    rox := server.NewRox()

    rox.RegisterWithEmptyNamespace(flags)

    log.Println("Initializing Feature Flags...")
    select {
    case <-rox.Setup(os.Getenv("FM_KEY"), options):
        log.Println("Feature Flags initialized successfully")
    case <-time.After(5 * time.Second):
        log.Println("Warning: Feature Flags timed out, using defaults")
    }

    app := NewApp(config, flags)

    server := &http.Server{
        Addr:              ":" + config.Port,
        Handler:           withCORS(config.AllowedOrigin, app.routes()),
        ReadHeaderTimeout: 5 * time.Second,
    }

    log.Printf("Theme Park API listening on %s", server.Addr)
    log.Printf("API endpoints: /api/health, /api/flags, /api/events, /api/promotions, /api/tickets/purchase, /api/reviews")
    if err := server.ListenAndServe(); err != nil {
        log.Fatal(err)
    }
}
----

**What we added:**

* `_ = godotenv.Load()` - Loads environment variables from `backend/.env` file
** The `_` ignores errors (file may not exist in production environments)
** Must be called **before** `LoadConfig()` or `os.Getenv("FM_KEY")`
** Reads your `FM_KEY` from `.env` and makes it available to `os.Getenv()`
* `server.NewRoxOptions()` - Creates SDK configuration
** `ImpressionHandler` - **Required for impression tracking in CloudBees dashboard**
** By default, the SDK does not automatically report impression data
** The handler is called every time a flag is evaluated
** Logs flag name and value for debugging
** In production, you can send this data to analytics or monitoring tools
* `server.NewRox()` - Creates SDK client
* `rox.RegisterWithEmptyNamespace(flags)` - Registers your flags with CloudBees
* `rox.Setup(os.Getenv("FM_KEY"), options)` - Connects to CloudBees using your SDK key from `.env`

[IMPORTANT]
====
**Impression Handler is Required for Dashboard Metrics**

Without the `ImpressionHandler`, the CloudBees dashboard will not show impression data (total impressions, unique users, variation distribution). The SDK does not automatically send this data - you must explicitly configure the handler to enable impression tracking in the Feature Management dashboard.
====
* `select` statement with timeout - Waits up to 5 seconds for connection
* `app := NewApp(config, flags)` - Passes flags to your application (note: we added `flags` parameter)

[IMPORTANT]
====
The `godotenv.Load()` call is **critical** for local development. Without it, Go cannot read the `.env` file and `FM_KEY` will be empty, causing the SDK connection to fail with "Invalid rollout apikey" error.
====

Save the file.

==== Step 4 - Update App Struct and Constructor

The `App` struct needs a `flags` field, and `NewApp()` needs to accept the flags parameter. Open `backend/handlers.go` and find:

[source,go]
----
type App struct {
    config Config
}

func NewApp(config Config) *App {
    return &App{
        config: config,
    }
}
----

Replace with:

[source,go]
----
type App struct {
    config Config
    flags  *Flags
}

func NewApp(config Config, flags *Flags) *App {
    return &App{
        config: config,
        flags:  flags,
    }
}
----

**What changed:**
1. Added `flags *Flags` field to the `App` struct
2. Added `flags *Flags` parameter to `NewApp()` function
3. Set `flags: flags` in the return statement

Save the file.

== Advanced Flag Patterns

=== Understanding CloudBees Properties

CloudBees uses two types of properties for feature flag targeting:

==== Built-in Properties

Built-in properties (prefixed with `rox.`) are automatically provided by the SDK:

* `rox.platform` - Platform name (e.g., "Go")
* `rox.now` - Current timestamp
* `rox.app_release` - Application version

NOTE: `rox.distinct_id` is only available in client-side SDKs (JavaScript browser, iOS, Android). Server-side SDKs like Go **do not generate** this property.

==== Custom Properties

Custom properties are values you define and pass in your code:

[source,go]
----
ctx := context.NewContext(map[string]interface{}{
    "email": "user@example.com",  // Custom property
    "plan":  "premium",
})
----

**For percentage rollouts with server-side SDKs:**

. Create the custom property in CloudBees UI (e.g., name: `email`)
. Pass it in your code context (e.g., `"email": userEmail`)
. Configure the flag's stickiness property to use your custom property
. Property names must match exactly between CloudBees and your code

The following patterns demonstrate how to use properties for advanced targeting and rollout strategies.

Refer to: https://docs.cloudbees.com/docs/cloudbees-feature-management/latest/properties[CloudBees Properties Documentation]

[IMPORTANT]
====
**User Context Required for Advanced Patterns**

For percentage rollouts and user targeting to work correctly, you **must pass user context** to flag evaluations. The backend already passes user context in the `/api/events` endpoint you updated in Step 5.

When you configured `handleEvents()`, you added:

[source,go]
----
email := r.URL.Query().Get("email")
roxContext := context.NewContext(map[string]interface{}{
    "email": email,  // Property name must match CloudBees custom property
})
// Then pass roxContext to IsEnabled():
if app.flags.ShowNewEvent.IsEnabled(roxContext) {
----

This ensures **the same user always gets the same flag value** (sticky rollout). The property name (`"email"`) must match the custom property you'll create in CloudBees for stickiness to work. Without user context, flags would evaluate randomly on each request, causing inconsistent experiences.

===== Understanding Context in Go SDK

The `context.NewContext()` function creates a context object for flag evaluation:

[source,go]
----
import "github.com/rollout/rox-go/v5/core/context"

// Create context with user properties
ctx := context.NewContext(map[string]interface{}{
    "email": "[email protected]",
})

// Evaluate flag with context
isEnabled := flags.ShowNewEvent.IsEnabled(ctx)
----

**How it works:**

. SDK receives context with your custom properties (e.g., `{"email": "user3@example.com"}`)
. CloudBees uses the configured stickiness property to extract the identifier
. SDK hashes the identifier value to a number between 0-1
. The hash determines which group the user falls into (e.g., < 0.20 = 20% group)
. Same identifier always produces the same hash (sticky behavior)

**Key Points:**

* Context properties are key-value pairs you define
* Property values come from your application (query parameters, sessions, etc.)
* CloudBees uses the stickiness property for consistent user experiences
* The property name in CloudBees must match the property name in your code

For more details, refer to: https://docs.cloudbees.com/docs/cloudbees-feature-management/latest/percentage-rollouts[CloudBees Percentage Rollouts]

// maintenanceMiddleware checks for maintenance mode and blocks requests if active
func (app *App) maintenanceMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        // Allow health check and flags endpoint even during maintenance
        if r.URL.Path == "/api/health" || r.URL.Path == "/api/flags" {
            next.ServeHTTP(w, r)
            return
        }

        // Check maintenance mode flag
        if app.flags.MaintenanceMode.IsEnabled(nil) {
            writeJSON(w, http.StatusServiceUnavailable, map[string]interface{}{
                "status":  "maintenance",
                "message": "System is currently under maintenance. Please check back soon.",
                "retry_after": 300, // Suggest retry after 5 minutes
            })
            return
        }

        next.ServeHTTP(w, r)
    })
}

== Configuration as Code (CasC)

=== Overview

Configuration as Code allows you to manage feature flags, target groups, and custom properties using YAML files in a Git repository. Instead of creating flags manually through the CloudBees UI, you define them in code and CloudBees syncs them automatically.

**Benefits of CasC**:

* **Version Control**: All flag changes tracked in Git
* **Code Review**: Flag configurations go through PR review process
* **Repeatability**: Replicate configurations across environments
* **Disaster Recovery**: Git is source of truth for all flags
* **Scale**: Manage hundreds of flags efficiently
* **Automation**: Integrate with CI/CD pipelines

**When to Use CasC**:

* Managing 10+ flags across multiple environments
* Teams requiring approval workflows for flag changes
* Organizations with strict change management requirements
* Production deployments requiring audit trails

**When to Use UI**:

* Learning feature management concepts
* Quick experiments or prototypes
* One-off flag changes during incidents
* Viewing real-time flag evaluation metrics

[NOTE]
====
CasC is **bidirectional**. Changes made in the UI sync back to Git, and changes pushed to Git sync to the UI. This allows teams to use both methods interchangeably.
====

==== Step 1 - Create First Flag Definition

Create the directory structure and flag definition file:

[source,console]
----
$ mkdir -p .cloudbees/casc/feature-management/flags
$ mkdir -p .cloudbees/casc/feature-management/flag-configurations/production
----

Create `.cloudbees/casc/feature-management/flags/default.DemoFeature.yaml`:

[source,yaml]
----
---
apiVersion: featuremanagement.cloudbees.io/v1alpha1
kind: flag

flag: default.DemoFeature
flagType: boolean
description: "My first CasC-managed feature flag"
labels:
  - casc-demo
  - tutorial

availableValues:
  - "true"
  - "false"

isPermanent: false
----

**Format Notes**:

* `flag: default.DemoFeature` - Flag name with `default.` namespace prefix
* `flagType: boolean` - Flag type (boolean, string, number)
* `availableValues` - Must list `"true"` first, then `"false"` (UI-generated pattern)
* `isPermanent: false` - Temporary flag that should be removed after rollout

==== Step 2 - Create Flag Configuration

Create `.cloudbees/casc/feature-management/flag-configurations/production/default.DemoFeature.config.yaml`:

[source,yaml]
----
---
apiVersion: featuremanagement.cloudbees.io/v1alpha1
kind: flag-configuration

flag: default.DemoFeature

enabled: true

stickinessProperty: rox.distinct_id

defaultValue: false

conditions: []
----

**Critical Format Requirements**:

* **Use `defaultValue:` NOT `value:`** - The `value` field does not sync from Git to UI
* **Include `stickinessProperty`** - Required even for simple flags
* **Include `conditions: []`** - Required even if empty
* **File naming**: `default.DemoFeature.config.yaml` - Must use `.config.yaml` extension

[IMPORTANT]
====
**Common Mistake**: Using `value: false` instead of `defaultValue: false`. The `value` field is used for condition outcomes, not the default flag value. CloudBees will silently ignore configurations with incorrect field names.
====

=== Task 3 - Create Target Group via CasC

Target groups allow you to segment users and apply flag values to specific audiences.

==== Step 1 - Create Custom Property

Create `.cloudbees/casc/feature-management/properties/user.email.yaml`:

[source,yaml]
----
---
apiVersion: v1
kind: flag-properties

name: user.email
type: String
description: "User email address for targeting"
----

==== Step 2 - Create Target Group

Create `.cloudbees/casc/feature-management/target-groups/InternalTesters.yaml`:

[source,yaml]
----
---
apiVersion: featuremanagement.cloudbees.io/v1alpha1
kind: target-group

name: InternalTesters
description: "Internal team members for early testing"

conditions:
  - property: user.email
    operator: regex
    operand: ".*@mycompany\\.com$"
----

**Available Operators by Property Type**:

**String operators**:

* `regex` - Regular expression match (requires `operand`)
* `is-undefined` - Property not set
* `in-array` - Value in list (requires `operand` as array)

**Boolean operators**:

* `is-true` - Property value is true
* `is-false` - Property value is false
* `is-undefined` - Property not set

**Number operators**:

* `gt` - Greater than (requires `operand`)
* `gte` - Greater than or equal
* `lt` - Less than
* `lte` - Less than or equal
* `eq` - Equal to
* `semver-gte` - Semantic version greater than or equal

==== Step 3 - Use Target Group in Flag

Update `.cloudbees/casc/feature-management/flag-configurations/production/default.DemoFeature.config.yaml`:

[source,yaml]
----
---
apiVersion: featuremanagement.cloudbees.io/v1alpha1
kind: flag-configuration

flag: default.DemoFeature

enabled: true

stickinessProperty: rox.distinct_id

conditions:
  - group:
      name: InternalTesters
    value: true

defaultValue: false
----

This configuration means:

* Users matching **InternalTesters** group get `true`
* All other users get `false` (defaultValue)

=== Task 5 - Best Practices for CasC

==== Workflow Pattern: UI First, CasC Second

**Recommended Workflow**:

. **Create flag in UI** - Use CloudBees UI to create initial flag
. **Verify flag works** - Test in development environment
. **Copy generated YAML** - CloudBees generates correct YAML format
. **Add to Git** - Copy YAML to repository
. **Use Git going forward** - All future changes via Git/PR

**Why This Works**:

* UI generates correct YAML format automatically
* Avoids syntax errors and format issues
* Provides working template to build from
* Eliminates guesswork about required fields

[IMPORTANT]
====
**Critical Format Tip**: If your flag configurations aren't syncing, create the flag in the UI first, then inspect the generated YAML in your repository after CloudBees syncs it. The UI-generated YAML is always correct. Use it as your template.
====

==== Field Naming Reference

**Common Mistakes**:

[source,yaml]
----
# ❌ WRONG - Will not sync
flag-configuration:
  value: false                    # Wrong field name
  stickinessProperty: email       # Wrong property format

# ✅ CORRECT
flag-configuration:
  defaultValue: false             # Correct field name
  stickinessProperty: rox.distinct_id  # Built-in property
----

**Percentage Splits**:

[source,yaml]
----
# ❌ WRONG - Missing 'option' field
defaultValue:
  - percentage: 50
    value: true                   # Wrong field name

# ✅ CORRECT
defaultValue:
  - percentage: 50
    option: true                  # Correct field name
----

==== File Naming Conventions

**Flag Definitions** (cross-environment):

* Location: `.cloudbees/casc/feature-management/flags/`
* Naming: `<namespace>.<FlagName>.yaml`
* Examples:
** `default.ShowNewEvent.yaml`
** `default.EnablePayments.yaml`

**Flag Configurations** (per-environment):

* Location: `.cloudbees/casc/feature-management/flag-configurations/<environment>/`
* Naming: `<namespace>.<FlagName>.config.yaml`
* Examples:
** `flag-configurations/production/default.ShowNewEvent.config.yaml`
** `flag-configurations/dev/default.EnablePayments.config.yaml`

**Target Groups**:

* Location: `.cloudbees/casc/feature-management/target-groups/`
* Naming: `<GroupName>.yaml`
* Examples:
** `BetaUsers.yaml`
** `InternalTesters.yaml`

**Custom Properties**:

* Location: `.cloudbees/casc/feature-management/properties/`
* Naming: `<namespace>.<propertyName>.yaml`
* Examples:
** `user.email.yaml`
** `user.subscription_tier.yaml`

==== Issue: Flag Not Syncing from Git to UI

**Symptoms**:

* Pushed YAML changes to Git
* No CloudBees bot commit appears
* Flag not visible in UI

**Diagnosis Steps**:

. Check GitHub App permissions:
** Navigate to GitHub organization settings

** Verify repository is included in app installation

. Validate YAML syntax:
+
[source,console]
----
$ yamllint .cloudbees/casc/feature-management/
----

. Check CloudBees integration status:
** Navigate to **Settings > Configuration as Code**
** Verify repository shows **Connected**
** Check for error messages

. Verify directory structure:
+
[source,console]
----
$ tree .cloudbees/
.cloudbees/
└── casc
    └── feature-management
        ├── flags
        ├── flag-configurations
        │   └── production
        ├── target-groups
        └── properties
----

. Check for required fields:
** `apiVersion: featuremanagement.cloudbees.io/v1alpha1`
** `kind: flag` or `kind: flag-configuration`
** `flag:` field present and matches across definition and configuration

**Common Causes**:

* Missing `.config.yaml` extension on configuration files
* Incorrect `apiVersion` (must be `featuremanagement.cloudbees.io/v1alpha1`)
* Environment folder name doesn't match CloudBees environment
* Using `value:` instead of `defaultValue:`

==== Issue: UI Changes Not Syncing to Git

**Symptoms**:

* Changed flag value in UI
* No CloudBees bot commit in repository

**Solutions**:

. Verify bidirectional sync is enabled:
** Navigate to **Settings > Configuration as Code**
** Ensure **Sync UI changes to Git** is enabled

. Check GitHub App write permissions:
** App must have write access to push commits

. Wait for sync interval:
** UI-to-Git sync can take 1-2 minutes
** Check for commits after waiting

==== Issue: Merge Conflicts in CasC Files

**Symptoms**:

* Pull request has conflicts in `.cloudbees/casc/` directory
* CloudBees bot commit conflicts with manual changes

**Resolution**:

When both UI and Git changes happen simultaneously:

[source,console]
----
$ git pull
# Conflict in flag configuration file

$ git status
both modified: .cloudbees/casc/feature-management/flag-configurations/production/default.MyFlag.config.yaml
----

**Resolution Strategy**:

. Accept CloudBees bot version (UI is source of truth):
+
[source,console]
----
$ git checkout --theirs .cloudbees/casc/feature-management/flag-configurations/production/default.MyFlag.config.yaml
$ git add .
$ git commit -m "Resolve conflict: accept UI changes"
----

. OR: Accept your version (Git is source of truth):
+
[source,console]
----
$ git checkout --ours .cloudbees/casc/feature-management/flag-configurations/production/default.MyFlag.config.yaml
$ git add .
$ git commit -m "Resolve conflict: use Git version"
$ git push
----

CloudBees will sync your pushed version back to the UI.

**Prevention**:

* Choose one path for changes: either UI or Git, not both simultaneously
* Coordinate flag changes with team members
* Use branch protection rules requiring PR reviews

==== Issue: Silent Rejection of Configuration

**Symptoms**:

* YAML file pushed successfully
* CloudBees bot commits appear
* BUT flag configuration doesn't match YAML in UI

**Common Causes**:

CloudBees silently ignores invalid configurations. Check for:

. **Wrong field names**:
+
[source,yaml]
----
# CloudBees ignores this
value: false              # Should be defaultValue: false
----

. **Missing required fields**:
+
[source,yaml]
----
# Missing stickinessProperty and conditions
flag: default.MyFlag
enabled: true
defaultValue: false
# Will not apply - missing required fields
----

. **Invalid percentage sum**:
+
[source,yaml]
----
# Percentages must sum to 100
defaultValue:
  - percentage: 50
    option: true
  - percentage: 40        # Only 90 total - invalid
    option: false
----

. **Operator requires operand**:
+
[source,yaml]
----
# regex operator requires operand
conditions:
  - property: email
    operator: regex
    # Missing: operand: ".*@company\\.com$"
----

**Validation Method**:

After pushing, compare UI to your YAML:

. Create flag in UI
. Let CloudBees sync to Git
. Compare generated YAML to your YAML
. Look for field differences

=== Task 7 - API Validation (Advanced)

For automated testing of CasC configurations, use the CloudBees API to validate flags:

==== Step 1 - Get API Credentials

. Navigate to **Settings > API Keys**
. Create new API key with **Read** permissions
. Save key as environment variable:
+
[source,console]
----
$ export CLOUDBEES_API_KEY="your-api-key"
$ export CLOUDBEES_API_URL="https://api.cloudbees.io/v2"
----

==== Step 2 - Create Validation Script

Create `scripts/validate-flags.sh`:

[source,bash]
----
#!/bin/bash
set -e

FLAG_NAME="$1"
ENVIRONMENT="${2:-production}"

echo "Validating flag: $FLAG_NAME in $ENVIRONMENT"

# Fetch flag configuration
RESPONSE=$(curl -s -H "Authorization: Bearer ${CLOUDBEES_API_KEY}" \
  "${CLOUDBEES_API_URL}/flags/${FLAG_NAME}/environments/${ENVIRONMENT}")

# Check flag exists
if echo "$RESPONSE" | jq -e '.name' > /dev/null; then
  echo "✅ Flag exists in CloudBees"
else
  echo "❌ Flag not found in CloudBees"
  exit 1
fi

# Check enabled status
ENABLED=$(echo "$RESPONSE" | jq -r '.enabled')
echo "Flag enabled: $ENABLED"

# Check default value
DEFAULT=$(echo "$RESPONSE" | jq -r '.defaultValue')
echo "Default value: $DEFAULT"

echo "✅ Validation complete"
----

Make script executable:

[source,console]
----
$ chmod +x scripts/validate-flags.sh
----

==== Step 3 - Run Validation

[source,console]
----
$ ./scripts/validate-flags.sh default.DemoFeature production
----

Expected output:

----
Validating flag: default.DemoFeature in production
✅ Flag exists in CloudBees
Flag enabled: true
Default value: false
✅ Validation complete
----

==== Step 4 - Integrate with CI/CD

Add validation to your CI/CD pipeline:

[source,yaml]
----
# Example: GitHub Actions workflow
name: Validate CasC Configuration

on:
  pull_request:
    paths:
      - '.cloudbees/casc/feature-management/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Wait for CloudBees sync
        run: sleep 60

      - name: Validate flags
        env:
          CLOUDBEES_API_KEY: ${{ secrets.CLOUDBEES_API_KEY }}
        run: |
          for flag in .cloudbees/casc/feature-management/flags/*.yaml; do
            FLAG_NAME=$(yq '.flag' "$flag")
            ./scripts/validate-flags.sh "$FLAG_NAME"
          done
----

This ensures all CasC changes are validated before merging.

== Best Practices

=== Naming Conventions

Use consistent, descriptive flag names:

**Format**: `<area>-<feature>-<description>`

**Examples**:

* `ui-dashboard-dark-mode`
* `api-payment-provider-stripe`
* `ops-system-maintenance-mode`
* `exp-pricing-discount-test`

**Guidelines**:

* Use lowercase with hyphens
* Be descriptive but concise
* Include feature area prefix
* Avoid version numbers

=== Flag Lifecycle Management

**Temporary Flags** (most feature flags):

* Create with end date in mind
* Remove after full rollout
* Archive in CloudBees
* Clean up code

**Permanent Flags** (operations & configuration):

* Mark as permanent in CloudBees
* Document purpose
* Review quarterly
* Examples: kill switches, regional settings

**Cleanup Cadence**:

* Weekly: Check for stale flags
* Monthly: Review flag inventory
* Quarterly: Archive unused flags

=== Security Considerations

**SDK Key Management**:

* Never commit SDK keys to version control
* Use environment variables or secrets management
* Rotate keys if exposed
* Use environment-specific keys

**Access Control**:

* Restrict Production flag changes to authorized users
* Enable approval workflows
* Audit flag change history
* Use RBAC for team permissions

**Network Security**:

* Ensure TLS for CloudBees communication
* Allow outbound HTTPS to `cloudbees.io`
* Monitor for certificate issues

=== Performance Optimization

**SDK Configuration**:

[source,go]
----
options := server.NewRoxOptions(server.RoxOptionsBuilder{
    FetchInterval:     60 * time.Second,    // Backup polling interval (default: 60s, minimum: 30s)
    ImpressionHandler: customHandler,       // Custom impression processing
})
----

[NOTE]
====
The `FetchInterval` is the backup polling frequency. CloudBees uses Server-Sent Events (SSE) for instant flag updates, so configuration changes typically apply immediately without waiting for the next poll.
====

**Best Practices**:

* Cache flag evaluations in performance-critical paths
* Batch flag checks when possible
* Use default values strategically
* Monitor SDK performance metrics

=== Code Practices

**Good - Clear Boundaries**:

[source,go]
----
if app.flags.EnableNewFeature.IsEnabled(ctx) {
    return handleNewFeature(req)
}
return handleLegacyFeature(req)
----

**Bad - Scattered Checks**:

[source,go]
----
result := process()
if flags.NewFeature {
    result = transform(result)
}
if flags.NewFeature {
    return formatNew(result)
}
return formatOld(result)
----

== Troubleshooting

=== Common Issues

==== Issue: Flag Not Evaluating Correctly

**Symptoms**:

* Flag always returns default value
* Changes in CloudBees not reflected

**Diagnosis**:

[source,console]
----
$ cd backend
$ FM_KEY=your-key go run .
----

Check logs for:

----
✅ Good: Feature Flags initialized successfully
❌ Bad: Feature Flags timed out, using defaults
----

**Solutions**:

* Verify `FM_KEY` is correct (check for trailing whitespace)
* Test network connectivity: `curl https://cloudbees.io`
* Check firewall allows outbound HTTPS
* Ensure SDK key is for correct environment

==== Issue: Targeting Rules Not Working

**Symptoms**:

* All users get same variation
* Percentage rollout not sticky

**Diagnosis**:

Check user context is passed:

[source,go]
----
roxContext := context.NewContext(map[string]interface{}{
    "email": email,  // Must match custom property in CloudBees
})
----

**Solutions**:

* Ensure `email` property is set in context and matches the custom property in CloudBees
* Verify the **Stickiness property** in CloudBees is set to your custom property (not `rox.distinct_id`)
* Verify user identifier is stable across sessions
* Check CloudBees rules match property names exactly
* Test with different user identifiers

==== Issue: Percentage Rollout Not Sticky

**Symptoms**: Same user sees different flag values on each refresh

**Common Causes**:

* Stickiness property in CloudBees doesn't match property name in code
* Custom property not created in CloudBees UI
* Empty or null context values being passed

**Solutions**:

* Verify custom property is created in CloudBees UI
* Ensure stickiness property setting matches your code property name exactly
* Confirm context contains valid, non-empty values
* Wait a few seconds for SSE notification (typically instant) after configuration changes

==== Issue: Impressions Not Recording

**Symptoms**:

* Flag impressions tab shows no data
* Metrics not updating

**Diagnosis**:

Enable SDK debug logging and check for impression errors.

**Solutions**:

* Verify SDK initialized successfully
* Check network connectivity
* Ensure impression endpoint not blocked
* Review SDK logs for errors

==== Issue: Performance Degradation

**Symptoms**:

* Slow request processing
* High latency

**Diagnosis**:

* Too many flag evaluations per request
* SDK network calls blocking

**Solutions**:

* Cache flag values when appropriate
* Batch flag evaluations
* Use local evaluation mode
* Review flag evaluation patterns

== Common Mistakes to Avoid

Based on consultant feedback, here are frequent pitfalls and how to avoid them:

=== Mistake 1: Forgetting to Return to Repository Root

**Problem**: Commands fail because you're in the wrong directory.

**Example**:
[source,console]
----
$ cd backend
$ go run .
# Later...
$ cd frontend  # Error! You're already in backend
----

**Solution**: Always check your location with `pwd` and navigate from repository root:
[source,console]
----
$ pwd
/Users/yourname/dup-theme-park-webapp

$ cd backend
# Do backend work
$ cd ..

$ pwd
/Users/yourname/dup-theme-park-webapp

$ cd frontend
# Do frontend work
----

=== Mistake 2: Using Wrong Branch

**Problem**: Starting with `feature-flags-complete` branch instead of `feature-flags-clean`.

**Consequence**: SDK already installed, nothing to learn!

**Solution**: Always verify branch at start:
[source,console]
----
$ git branch
* feature-flags-clean  ✅ Correct!
  main

# If on wrong branch:
$ git checkout feature-flags-clean
----

=== Mistake 3: Committing .env Files

**Problem**: Accidentally commit API keys to git repository.

**Security Risk**: Keys exposed publicly, must be rotated immediately.

**Solution**: Verify before every commit:
[source,console]
----
$ git status

# backend/.env should NOT appear in the list
# frontend/.env should NOT appear in the list

# If they appear:
$ git reset HEAD backend/.env
$ git reset HEAD frontend/.env
----

The `.gitignore` file already prevents this, but double-check!

=== Mistake 4: Not Keeping Both Terminals Open

**Problem**: Closing backend terminal while testing frontend.

**Consequence**: Frontend shows "Network Error" or "Cannot connect to API".

**Solution**: Use terminal tabs or split screen:
[source,console]
----
# Terminal 1 (must stay open):
$ cd backend && go run .
Theme Park API listening on :8080

# Terminal 2 (must stay open):
$ cd frontend && npm run dev
Local: http://localhost:5173/
----

Press `Ctrl+C` to stop, but keep both terminals alive!

=== Mistake 5: Copy-Pasting FM_KEY with Extra Whitespace

**Problem**: FM_KEY has leading/trailing spaces or newlines.

**Example**:
[source,properties]
----
FM_KEY= xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx    ← Extra spaces!
----

**Consequence**: SDK fails to initialize, flags return defaults.

**Solution**: Trim whitespace carefully:
[source,properties]
----
FM_KEY=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
----

No spaces before or after the key!

=== Mistake 6: Not Waiting for SDK Sync

**Problem**: Expecting immediate flag updates after changing in CloudBees.

**Reality**: The SDK uses Server-Sent Events (SSE) for instant notifications, typically updating within seconds. As a backup, the SDK polls every 60 seconds.

**Solution**: Wait a few seconds for SSE notification, or restart backend for guaranteed immediate update:
[source,console]
----
# Option 1: Wait (recommended)
# Make change in CloudBees
# Wait a few seconds (SSE is typically instant)
# Refresh browser
# If no update after 10 seconds, wait up to 60 seconds for polling

# Option 2: Restart (guaranteed immediate)
# Press Ctrl+C in backend terminal
$ go run .
----

=== Mistake 7: Editing Code on Wrong Branch

**Problem**: Making changes on `feature-flags-complete` instead of `clean`.

**Consequence**: Can't compare your work with reference implementation.

**Solution**: Check branch before every edit:
[source,console]
----
$ git branch
* feature-flags-clean  ✅ Safe to edit!

# If on wrong branch:
$ git stash  # Save your work
$ git checkout feature-flags-clean
$ git stash pop  # Restore your work
----
